| Код | Требования | Комментарий |
|-----|-----------|-------------|
| F | Функциональные (Functionality) | |
| F1 | На сайте новый клиент видит список доступных депозитов с актуальными ставками. Клиент может подать заявку на депозит, указав номер телефона и Ф. И. О. | Сайт — точка входа для новых клиентов. |
| F2 | После подачи заявки на сайте менеджер кол-центра связывается с клиентом и может предложить особые условия. | Заявка с сайта должна попадать в систему кол-центра. |
| F3 | Новый клиент после звонка из кол-центра должен прийти в отделение для идентификации (KYC). | Регуляторное требование — банк не может принимать новых клиентов без личной проверки документов. |
| F4 | В интернет-банке клиент видит список депозитов с актуальными и персонализированными ставками. Указав счёт и сумму, подаёт заявку на открытие депозита. | Интернет-банк — основной канал для существующих клиентов. Персонализированные ставки требуют интеграции с АБС и модулем расчёта ставок. |
| F5 | Операция открытия депозита в интернет-банке подтверждается с помощью СМС-кода. | Двухфакторная аутентификация. Функционал СМС в ядре системы подрядчика, но лучше реализовать силами банка. |
| F6 | Ставки по депозитам должны вестись централизованно, а не только в XLS-файлах. Доступ нужен бэк-офису депозитов и кредитов. | Архитектурно-значимое: влияет на выбор хранилища и интеграцию. Целесообразно реализовать в АБС. |
| F7 | На этапе MVP заявку обрабатывает менеджер бэк-офиса — подтверждает условия депозита в АБС. | MVP — ручная обработка. Полная автоматизация — цель на год. |
| F8 | Клиент получает СМС-уведомления после подтверждения ставки и после открытия депозита. | Требуется интеграция с СМС-шлюзом. Отправку СМС реализовать силами банка, не затрагивая ядро подрядчика. |
| F9 | Данные с сайта и из интернет-банка при передаче защищены шифрованием трафика (TLS/SSL). | Архитектурно-значимое: влияет на инфраструктуру и конфигурацию всех каналов. |
| F10 | Сотрудники депозитов и кредитов не должны иметь прямого доступа к данным друг друга. | Требование ИБ из текущего процесса. При централизации ставок в АБС — обеспечить ролевую модель доступа. |
| U | Удобство использования (Usability) | |
| U1 | Интерфейс работы должен быть максимально удобным для клиента. | Бизнес-требование: привлечение молодой аудитории. |
| U2 | Придерживаться системы дизайна компании. Пользователи должны видеть привычные цвета и брендированные элементы. | Архитектурно-значимое: единая дизайн-система для сайта и интернет-банка. Требует общей библиотеки UI-компонентов. |
| U3 | Подача заявки на сайте должна быть простой: клиент указывает минимум данных (телефон, Ф. И. О.). | Снижение порога входа для новых клиентов. |
| U4 | Открытие депозита в интернет-банке: выбор депозита, указание счёта и суммы, подтверждение по СМС. | Минимум шагов для существующего клиента. |
| R | Надёжность (Reliability) | |
| R1 | Все сервисы должны работать 24/7, доступность 99,9% (SLA). | Архитектурно-значимое: требует отказоустойчивой архитектуры, резервирования, мониторинга. Текущая инфраструктура ИБ не обеспечивает этого. |
| R2 | При сбое основного ЦОД сервисы интернет-банка переключаются на резервный ЦОД. | Архитектурно-значимое: необходим failover. Резервный ЦОД есть, но переключение не автоматизировано. |
| R3 | Избежать прямой работы интернет-банка с API АБС в новом процессе. | Архитектурно-значимое: БД АБС перегружена. Нужен промежуточный слой (очередь сообщений, кэш или отдельный сервис). |
| R4 | АБС масштабируется только вертикально из-за базы данных Oracle. | Архитектурно-значимое: ограничение необходимо учитывать при проектировании нагрузки. Аргумент в пользу промежуточного слоя. |
| R5 | Онлайн-подача заявок не должна ставить под угрозу работоспособность АБС и банка. | Изоляция нагрузки от онлайн-каналов и критичных банковских операций. |
| P | Производительность (Performance) | |
| P1 | Отклик по всем операциям в интернет-банке — в пределах миллисекунд. | Архитектурно-значимое: требует оптимизации запросов, кэширования, асинхронной обработки. Сейчас справочные данные грузятся больше секунды. |
| P2 | Равномерное горизонтальное масштабирование и распределение запросов между серверами, приложениями и ЦОД. | Архитектурно-значимое: требуется балансировщик нагрузки, горизонтальное масштабирование ИБ. |
| P3 | Справочные данные в интернет-банке должны загружаться быстрее (сейчас более 1 секунды). | Текущая проблема производительности. Решение — кэширование справочных данных. |
| S | Поддерживаемость (Supportability) | |
| S1 | Максимально использовать существующие технологии банка (MS SQL, Oracle). Новые технологии — совместимы с текущими платформами. | Архитектурно-значимое: ограничивает выбор стека. Снижает стоимость поддержки. |
| S2 | Желательно наличие экспертизы внутри банка по новым технологиям. | Снижение рисков. В команде АБС есть специалисты с Java-стеком. |
| S3 | Для очередей сообщений использовать Kafka. Текущая платформа ИБ (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka. | Архитектурно-значимое: для Kafka нужна модернизация платформы ИБ (миграция на новый .NET или микросервисы). |
| S4 | Рассмотреть перевод интернет-банка на микросервисную архитектуру в рамках задачи открытия депозитов. | Архитектурно-значимое: поэтапная миграция монолита. Независимое развитие, но сложнее инфраструктура. |
| S5 | Предусмотреть разработку документации для дальнейшего расширения системы. | Устойчивое развитие и онбординг новых сотрудников. |
| S6 | Функционал отправки СМС реализовать силами банка, не завися от доработок подрядчика ядра. | Архитектурно-значимое: снижение зависимости от подрядчика. |
| S7 | Система кол-центра имеет возможности для расширения, в команде АБС есть Java-специалисты для самостоятельных доработок. | Снижение зависимости от подрядчика кол-центра. |
| +R | + Ограничения (Restrictions) | |
| +R1 | Интернет-банк — монолит на ASP.NET MVC 4.5 / .NET Framework 4.5, СУБД MS SQL. Обновление ядра привязано к подрядчику. | Архитектурно-значимое: технологическое ограничение платформы. Ограничивает выбор библиотек и интеграций. |
| +R2 | АБС — десктопный клиент на Delphi, СУБД Oracle, логика на PL-SQL. Внутренняя разработка. | Архитектурно-значимое: ограничивает способы интеграции (нет REST API из коробки). Нужен интеграционный слой. |
| +R3 | Система кол-центра — платформа подрядчика (React.js, Java Spring Boot, PostgreSQL, микросервисная). Развёрнута в инфраструктуре банка. | Ограничение на самостоятельные доработки, но есть возможности для расширения. |
| +R4 | Сайт — собственная разработка банка на PHP и React.js. | Стек сайта отличается от интернет-банка. Возможна самостоятельная доработка. |
| +R5 | Партнёрский кол-центр — внешняя система другой компании. | Ограниченные возможности интеграции. |
| +R6 | Телеком-оператор — внешний оператор по отправке СМС. Взаимодействие через СМС-шлюз. | Внешняя зависимость. |
| +R7 | Банки не могут принимать новых клиентов без личной проверки документов (KYC). | Регуляторное ограничение: новые клиенты с сайта обязаны прийти в отделение. |
| +R8 | Сотрудники депозитов и кредитов не должны иметь доступа к данным друг друга. | Ограничение ИБ на архитектуру ролевой модели при централизации управления ставками. |

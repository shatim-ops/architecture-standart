### Нефункциональные требования

| № | Требование |
| :-: | :- |
| NFR-1 | Доступность всех сервисов 24/7, SLA 99,9%. Интернет-банк должен выдерживать отказ основного ЦОД с переключением на резервный. |
| NFR-2 | Избежать прямого взаимодействия интернет-банка с API АБС — БД АБС перегружена. Использовать промежуточный слой. |
| NFR-3 | Отклик по операциям в интернет-банке — в пределах миллисекунд. Справочные данные кэшировать для ускорения загрузки (сейчас более 1 сек). |
| NFR-4 | Горизонтальное масштабирование интернет-банка с балансировкой нагрузки между серверами и ЦОД. АБС — только вертикальное масштабирование. |
| NFR-5 | Шифрование трафика (TLS/SSL) для сайта и интернет-банка — передаётся чувствительная финансовая и персональная информация. |
| NFR-6 | Разграничение доступа: сотрудники депозитов и кредитов не должны иметь доступа к данным друг друга. |
| NFR-7 | Максимальное использование существующих технологий банка (MS SQL, Oracle). Совместимость новых технологий с текущими платформами. |
| NFR-8 | Использование Kafka для асинхронного обмена сообщениями. Требует модернизации платформы интернет-банка (текущая ASP.NET MVC 4.5 несовместима). |
| NFR-9 | Единая дизайн-система для сайта и интернет-банка. Брендированные элементы и привычные цвета. |
| NFR-10 | Документирование архитектуры для дальнейшего расширения системы. |

### Решение

Для реализации MVP предлагается выделить новый микросервис — Сервис депозитов, который станет промежуточным слоем между интернет-банком и АБС. Это позволяет решить ключевую проблему: изолировать перегруженную БД АБС от прямых запросов из онлайн-каналов.

Логика принятия решений:

1. Промежуточный сервис (Сервис депозитов). БД АБС перегружена (NFR-2), АБС масштабируется только вертикально (NFR-4), при этом требуется SLA 99,9% (NFR-1). Прямое взаимодействие интернет-банка с АБС недопустимо. Поэтому вводится отдельный Сервис депозитов, который:
   - принимает заявки от интернет-банка и сайта;
   - кэширует актуальные ставки (NFR-3);
   - асинхронно взаимодействует с АБС через Kafka (NFR-8);
   - имеет собственную базу данных для хранения заявок и кэша ставок.

2. Микросервисная архитектура для новой функциональности. Текущий интернет-банк — монолит на ASP.NET MVC 4.5. Полный рефакторинг нецелесообразен. Поэтому новая функциональность (открытие депозитов) выделяется в отдельный микросервис. Интернет-банк обращается к нему по REST API. Это позволяет:
   - не затрагивать ядро монолита подрядчика;
   - использовать Kafka (которая несовместима с текущей платформой ИБ);
   - независимо масштабировать сервис депозитов.

3. Kafka как шина сообщений. Используется для асинхронной связи между Сервисом депозитов и АБС. Это обеспечивает:
   - развязку компонентов;
   - устойчивость к пиковым нагрузкам (сообщения буферизируются);
   - надёжную доставку заявок даже при временной недоступности АБС.
   В команде АБС есть специалисты с Java-стеком, что упрощает разработку Kafka-потребителя на стороне АБС.

4. Технологический стек Сервиса депозитов. Java Spring Boot + PostgreSQL — совместим с Kafka, экспертиза уже есть в команде АБС, а также аналогичный стек используется в системе кол-центра. Это соответствует требованию использовать технологии с существующей экспертизой (NFR-7).

5. Отправка СМС. Сервис депозитов самостоятельно взаимодействует с СМС-шлюзом телеком-оператора для отправки уведомлений. Это позволяет не зависеть от доработок подрядчика ядра интернет-банка.

6. Интеграция сайта. Сайт (PHP + React.js) передаёт заявки в систему кол-центра через её API. Система кол-центра (Java Spring Boot, микросервисная) имеет возможности для расширения.

7. Отказоустойчивость. Сервис депозитов разворачивается в обоих ЦОД. Балансировщик нагрузки распределяет запросы между ЦОД. В случае сбоя основного ЦОД трафик переключается на резервный (NFR-1, NFR-2).

8. Централизация ставок. Функциональность расчёта и хранения ставок переносится из XLS-файлов в АБС. Сервис депозитов получает обновлённые ставки через Kafka и кэширует их в своей БД.

### Альтернативы

1. Прямая интеграция интернет-банка с АБС через REST API. Отвергнута, так как БД АБС перегружена и не выдержит дополнительную нагрузку от онлайн-канала. Кроме того, АБС не имеет REST API из коробки (десктопный клиент на Delphi, логика на PL-SQL), что потребовало бы значительных доработок.

2. Реализация функциональности депозитов внутри монолита интернет-банка. Отвергнута, поскольку обновление ядра системы привязано к подрядчику, а текущая платформа (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka. Расширение монолита увеличит связанность и усложнит дальнейшее развитие.

3. Использование RabbitMQ вместо Kafka. Рассмотрена, но отвергнута: бизнес рекомендовал Kafka на перспективу для возможного масштабирования на другие продукты. Kafka лучше подходит для сценариев с высокой пропускной способностью и гарантированной доставкой.

4. Синхронное взаимодействие Сервиса депозитов с АБС. Отвергнута из-за вертикального ограничения масштабирования АБС и необходимости буферизации запросов при пиковых нагрузках.

### Недостатки, ограничения, риски

1. Усложнение инфраструктуры. Введение Kafka и нового микросервиса увеличивает сложность эксплуатации. Необходимы компетенции по настройке и мониторингу Kafka-кластера. Митигация: использование существующих Java-компетенций в команде АБС, а также внедрение мониторинга.

2. Eventual consistency. Асинхронное взаимодействие через Kafka означает, что данные в Сервисе депозитов и АБС не всегда синхронны. Ставки могут обновляться с задержкой. Митигация: для MVP приемлема задержка в минуты. В будущем можно уменьшить интервал обновления.

3. Зависимость от подрядчика интернет-банка. Даже при выделении микросервиса, интернет-банк (монолит) должен уметь маршрутизировать запросы к новому сервису. Это может потребовать минимальных доработок в монолите. Митигация: изменения минимальны — добавление проксирования запросов или API Gateway.

4. Дополнительные затраты на два ЦОД. Развёртывание сервиса и Kafka в двух ЦОД увеличивает стоимость инфраструктуры. Митигация: это необходимо для обеспечения SLA 99,9%.

5. Ограничение масштабирования АБС. АБС остаётся узким местом, масштабируется только вертикально. При значительном росте количества заявок может потребоваться более глубокая оптимизация на стороне АБС. Митигация: Kafka буферизирует нагрузку, обработка заявок в АБС происходит асинхронно.

6. MVP предполагает ручную обработку. На этапе MVP менеджер бэк-офиса вручную обрабатывает заявки в АБС, что ограничивает пропускную способность. Митигация: это временное решение, автоматизация запланирована на следующий этап (через год).

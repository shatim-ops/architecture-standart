### Решение

Для решения задачи передачи ставок в кол-центры используется инфраструктура Kafka, спроектированная в задании 3 для MVP открытия депозитов. Ставки из АБС уже публикуются в Kafka через Kafka-адаптер АБС. Необходимо добавить двух новых потребителей:

1. Потребитель в системе кол-центра банка. Система кол-центра (Java Spring Boot, микросервисная) подключается как Kafka Consumer к топику ставок. Это естественное расширение: система построена на Java Spring Boot, Kafka-клиенты для Java хорошо поддерживаются. В команде АБС есть Java-специалисты, которые могут выполнить эту доработку без привлечения подрядчика. Полученные ставки сохраняются в БД системы кол-центра (PostgreSQL) и отображаются операторам в интерфейсе.

2. SFTP-экспорт для партнёрского кол-центра. Партнёрский кол-центр — внешняя система, не имеющая возможности API-интеграции. Единственный доступный протокол — SFTP. Поэтому в Сервисе депозитов реализуется компонент экспорта: при получении обновлённых ставок из Kafka сервис формирует файл (CSV) и загружает его на SFTP-сервер банка. Партнёрский кол-центр забирает файл по расписанию. SFTP-сервер разворачивается в инфраструктуре банка.

Логика выбора решения:

- Kafka как единая шина для распространения ставок. Ставки уже публикуются в Kafka из АБС (решение задания 3). Добавление новых потребителей не требует изменений в АБС или Kafka-адаптере — только подписка на существующий топик. Это паттерн pub/sub, обеспечивающий слабую связанность.

- Доработка системы кол-центра силами банка. Система кол-центра имеет возможности для расширения. Java-специалисты из команды АБС могут реализовать Kafka Consumer и экран отображения ставок, минимизируя зависимость от подрядчика.

- SFTP для партнёрского кол-центра. Партнёр не поддерживает API — это жёсткое ограничение. SFTP — стандартный протокол для безопасной передачи файлов. Файлы формируются в Сервисе депозитов, который уже имеет актуальный кэш ставок.

### Альтернативы

1. Прямой REST API из системы кол-центра к Сервису депозитов для получения ставок. Рассмотрена, но отвергнута: создаёт синхронную зависимость между системами. При недоступности Сервиса депозитов операторы кол-центра не смогут видеть ставки. Kafka обеспечивает асинхронную доставку и локальное хранение ставок в БД кол-центра.

2. Передача ставок в партнёрский кол-центр по электронной почте (как сейчас делается с XLS-файлами). Отвергнута: ненадёжно, ручной процесс, не масштабируется, нет гарантий доставки. SFTP автоматизирует процесс и обеспечивает подтверждение доставки.

3. Реализация отдельного API Gateway для партнёрского кол-центра. Отвергнута: партнёр явно заявил, что не может делать API-вызовы. Только файловый обмен через SFTP.

4. Хранение ставок для кол-центра в отдельной БД (вне системы кол-центра). Отвергнута: усложняет инфраструктуру без необходимости. Логичнее хранить ставки в БД самой системы кол-центра (PostgreSQL), так как операторы работают именно в этой системе.

### Недостатки, ограничения, риски

1. Eventual consistency ставок. Ставки в системе кол-центра и у партнёра обновляются асинхронно. Возможна ситуация, когда оператор видит устаревшие ставки. Митигация: допустимая задержка — до 15 минут. Ставки обновляются ежедневно, поэтому задержка в минуты приемлема.

2. Зависимость от SFTP-инфраструктуры. Сбой SFTP-сервера или сети приведёт к тому, что партнёрский кол-центр не получит обновлённые ставки. Митигация: retry-механизм в Сервисе депозитов, мониторинг SFTP-сервера, хранение последнего успешно отправленного файла.

3. Ограниченное влияние на партнёрский кол-центр. Банк не контролирует, как часто партнёр забирает файлы и как их использует. Митигация: SLA с партнёром на частоту проверки SFTP-сервера.

4. Доработка системы кол-центра силами банка, а не подрядчика. Есть риск внесения изменений, несовместимых с будущими обновлениями от подрядчика. Митигация: использовать механизмы расширения системы (плагины, отдельные микросервисы), не изменяя ядро.

5. Увеличение нагрузки на кол-центр. Маркетинговая кампания может привести к резкому росту звонков. Партнёрский кол-центр подключается для разгрузки, но требуется время на обучение операторов партнёра. Митигация: заблаговременная подготовка и тестирование процесса до запуска кампании.

### Список крупных задач по системам

АБС:
- Убедиться, что Kafka-адаптер АБС корректно публикует актуальные ставки в топик Kafka (реализовано в рамках задания 3).
- При необходимости доработать формат сообщений в Kafka для включения всех данных, нужных кол-центрам.

Сервис депозитов:
- Реализовать компонент экспорта ставок в файл (CSV/XLS).
- Реализовать SFTP-клиент для загрузки файлов на SFTP-сервер банка.
- Реализовать retry-механизм при сбоях SFTP.
- Настроить расписание формирования и отправки файлов.

Система кол-центра банка:
- Реализовать Kafka Consumer для подписки на топик ставок.
- Создать хранилище ставок в БД кол-центра (PostgreSQL).
- Разработать экран отображения актуальных ставок для операторов.
- Протестировать интеграцию.

Инфраструктура:
- Развернуть SFTP-сервер в инфраструктуре банка.
- Настроить доступ партнёрского кол-центра к SFTP-серверу (учётные записи, SSH-ключи).
- Настроить мониторинг SFTP-сервера и процесса передачи файлов.

Партнёрский кол-центр:
- Согласовать формат файла со ставками.
- Настроить автоматическое получение файлов с SFTP-сервера.
- Обучить операторов работе с новыми данными по ставкам.

Kafka:
- Убедиться, что топик ставок поддерживает нескольких потребителей (consumer groups).
- При необходимости настроить дополнительные consumer groups для системы кол-центра.
